# AI 規劃模式（/brainstorm）

把模糊的想法變成具體的計劃書。透過引導式問答，幫你在 AI 動手寫 code 之前，先釐清「到底要做什麼」。

> 致謝：核心理念參考自 [obra/superpowers](https://github.com/obra/superpowers)（brainstorming skill, MIT License），經過簡化與在地化改造。

---

<HARD-GATE>
在用戶確認計劃書之前，絕對不可以：
- 寫任何程式碼
- 建立任何檔案（計劃書本身除外）
- 執行任何實作動作
- scaffold 任何專案結構

不管任務看起來多簡單，都要先走完規劃流程。
一個待辦事項 App、一個簡單腳本、一個設定檔修改，全部都要。
計劃書可以很短（簡單任務幾句話就夠），但流程不能跳過。
</HARD-GATE>

## 輸入格式

- `/brainstorm 我想做一個記帳 App` — 帶主題啟動
- `/brainstorm` — 不帶主題，AI 會問你想做什麼

---

## 流程（按順序執行，不可跳步）

### Phase 0：確認溝通模式 ⭐ 最重要的第一步

**這是整個 brainstorm 流程的第一個問題，不可跳過。** 它決定了後續所有對話、假設列表、方案比較、計劃書的呈現方式。

用 **AskUserQuestion** 問：

> 你希望我用什麼樣的程度，跟你進行專案與需求的討論及展開？
>
> 1. 🟢 **小白模式**（推薦給非工程師）
>    我會盡量用白話講，少用技術名詞，重點放在「你會得到什麼」「有什麼風險」「未來可以怎麼延伸」。技術細節我自己處理。
>
> 2. 🟡 **半技術模式**
>    保留一些必要的技術名詞（檔名、套件名、API 名稱），但會搭配說明。適合有點概念但不想太深入細節的人。
>
> 3. 🔴 **工程師模式**
>    直接用技術詞彙溝通（檔案路徑、套件版本、架構名詞），假設你看得懂。最精簡、最快。
>
> 4. 🤔 **你決定就好**
>    AI 看情況自己判斷（會根據你後續的回答自動調整）。

將選擇記為 `COMMUNICATION_MODE`，**從這個 Phase 之後的每一句話都要套用**。

#### 🟢 小白模式的具體規則

當 `COMMUNICATION_MODE = 小白` 時：
- **盡可能減少技術名詞**：用「網頁的設定檔」而不是「config.js」
- **直接溝通成果**：每個方案都先說「做完你會看到什麼」
- **說明相關風險與延續性發展**：每個方案都附上「可能的風險」和「未來可以怎麼延伸」
- **假設列表**裡不要出現純技術名詞，改寫成成果導向
- **方案比較表**欄位：「做完長什麼樣」「優點」「缺點」「風險」「未來能延伸」「你需要做的事」

#### 🟡 半技術模式
- 保留檔名、套件名、API 名稱等關鍵詞
- 第一次出現專業名詞時加一句白話說明

#### 🔴 工程師模式
- 直接用技術詞彙，不需解釋
- 方案比較表可以有完整的技術細節

#### 🤔 自動判斷
- 預設先用半技術模式，根據用戶回答自動切換

### Phase 1：暖身確認

**Step 1-1：確認主題**

如果 `/brainstorm` 後面有寫主題，直接使用。
如果沒有，用 **AskUserQuestion** 問：「你想做什麼？用一句話描述就好。」

**Step 1-2：分類**

用 **AskUserQuestion** 問：

> 這比較像是哪一種？
> 1. 🆕 做一個新東西（新功能、新專案、新工具）
> 2. 🔧 解決一個問題（Bug、錯誤、某個東西壞了）
> 3. ✨ 改善現有的東西（優化、重構、讓它更好用）
> 4. 🤔 我還不確定（讓 AI 幫我釐清）

將選擇記為 PROJECT_TYPE。

---

### Phase 2：快速探索

**Step 2-1：掃描上下文（AI 自動執行）**

快速了解現況：
- 讀 CLAUDE.md（如果有）
- 看最近 5 筆 git commit（如果是 git repo）
- 用 Glob 掃描專案結構
- 如果主題提到特定檔案，找到並讀取

**Step 2-2：列出假設**

根據掃描結果，列出 3-5 個理解和假設。
⚠️ 必須套用 `COMMUNICATION_MODE`。小白模式不可出現純技術詞彙。

格式（半技術 / 工程師模式）：
```
根據你的描述，我先列出理解，你看看哪些需要修正：
1. 目標：...
2. 技術：...
3. 範圍：...
```

格式（小白模式）：
```
根據你的描述，我先講講我理解的方向：
1. 你想做什麼：...（白話）
2. 使用方式：...（白話）
3. 這次先做的功能：...
4. 未來可以延伸：...
```

然後用 **AskUserQuestion** 問：
> 以上的理解，有哪些需要修正？
> 1. ✅ 全部正確，繼續！
> 2. ✏️ 有些要修正
> 3. 🔄 大部分都不對，重新來

---

### Phase 3：深度釐清

依 PROJECT_TYPE，一次問一個問題釐清細節。

**規則**：
- 永遠用 AskUserQuestion
- 一次只問一個問題
- 優先選擇題，最後一個選項留給自由發揮
- 最多問 5 個問題
- 已確認的事不再問

**不耐煩偵測**：用戶出現「直接做吧」「你決定就好」或回答變很短 → 最多再問一個關鍵問題，說：「好，剩下的我根據目前的理解來規劃，有不對的地方再調整。」

---

### Phase 4：方案提議

提出 2-3 個具體方案，必須套用 COMMUNICATION_MODE。

格式（小白模式）：
```
## 三個做法比一比

| | 做法 A | 做法 B | 做法 C |
|:--|:--|:--|:--|
| 做完長什麼樣 | ... | ... | ... |
| 好處 | ... | ... | ... |
| 壞處 | ... | ... | ... |
| 可能的風險 | ... | ... | ... |
| 未來能延伸 | ... | ... | ... |
| 你需要做的事 | ... | ... | ... |

💡 我推薦做法 X，因為 [白話一句理由]。
```

如果任務很簡單，直接說「這個任務比較直接，我建議的做法是 [說明]，你覺得 OK 嗎？」

用 **AskUserQuestion** 讓用戶選方案。

---

### Phase 5：計劃書生成

**Step 5-1：撰寫計劃書**

格式：
```markdown
# [計劃名稱]

> 建立時間：YYYY-MM-DD HH:MM
> 類型：[新功能 / 問題修復 / 改善優化]
> 狀態：待執行

---

## 你要做的事（一句話版）

[白話說清楚這次要做什麼]

## 背景和動機

[為什麼要做？解決什麼問題？]

## 具體步驟

### Step 1：[步驟名稱]
- 做什麼：[具體動作]
- 產出：[這步做完會有什麼]
- 注意：[要小心的地方]

（3-8 個步驟）

## 預計成果

做完以後你會得到：
- [成果 1]
- [成果 2]

## 不包含在這次的範圍

- [明確列出這次不做的事]

## 可能遇到的風險

- [風險]：[怎麼處理]
```

**Step 5-2：Self-Review（AI 內部，不告知用戶）**

檢查：
1. 有無「TBD」「待確認」「之後再決定」？→ 直接填上
2. 步驟之間有無矛盾？→ 修正
3. 步驟超過 8 個？→ 考慮拆成多個計劃
4. 有無模糊指令（「適當處理」「做好測試」）？→ 改具體

**Step 5-3：存檔**

存到 `100_Todo/plans/`，檔名格式：`YYYY-MM-DD-[主題關鍵字].md`

告知用戶：「計劃書已存到 [路徑]。」

---

### Phase 6：下一步

用 **AskUserQuestion** 問：

> 計劃書好了！你想怎麼做？
> 1. 🚀 現在就開始執行！
> 2. 📝 我想先修改計劃書
> 3. 💾 先存著，之後再做

選 1 → 退出 brainstorm，按計劃書步驟執行。
選 2 → 讓用戶說明修改，改完後 self-review，再問一次。
選 3 → 結束，提醒計劃書位置。

---

## 護欄（Guardrails）

這些規則整個流程永遠生效：

- **不准寫 code**：確認計劃書前，一行都不能寫
- **不准跳步**：每個 Phase 都要過，可以快但不能跳
- **一次一個問題**：不能在同一則訊息裡問兩個以上的問題
- **用選項框**：能用 AskUserQuestion 就用
- **不要空洞附和**：不說「好問題」「這很有趣」，直接回應內容
- **挑戰假設**：如果範圍太大，直接說：「我建議先做 [核心部分]，之後再加其他的」
- **YAGNI 原則**：只留現在確定需要的，砍掉「以後可能用到」的功能
- **永遠遵守 COMMUNICATION_MODE**：Phase 0 確認後，所有輸出都要套用
- **全程繁體中文**：包括執行階段的所有對用戶可見文字。允許例外：內部推理、工具參數、程式碼本身
